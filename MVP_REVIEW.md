# 모두의 솔로몬 MVP 공개 전 검토 보고서

> 작성일: 2026-05-13  
> 기준: 실제 유저에게 공개 가능한 MVP 수준 달성을 위한 객관적 감사

---

## 전체 평가 요약

**현재 상태: "작동하는 프로토타입" 수준. 실사용 공개 불가.**

핵심 비즈니스 로직(블라인드 룰, 솔로몬 지수, 투표 제한)은 잘 설계되어 있으나, 게시글 작성이 실제로 동작하지 않는 치명적 버그가 존재하고, 실사용자 운영을 위한 최소 요건(신고, 약관, 비밀번호 찾기 등)이 누락되어 있다.

---

## 🔴 P0 — 서비스 자체가 동작하지 않는 버그 (즉시 수정)

### 1. 게시글 작성 API 데이터 불일치 버그

**발생 위치:** `app/post/create/page.tsx` ↔ `app/api/posts/route.ts`

클라이언트는 `options`를 객체 배열로 전송하지만 서버는 문자열 배열로 파싱한다. 결과적으로 게시글 작성이 항상 실패한다.

```typescript
// 클라이언트가 보내는 형태
options: [{ option_text: "선택지1", option_order: 1 }, ...]

// 서버가 기대하는 형태 (버그)
body.options.filter((o): o is string => typeof o === "string")
// → 객체는 전부 필터링되어 optionTexts = [], 400 에러 반환
```

**영향:** 게시글 작성 완전 불가. 서비스의 핵심 기능이 동작하지 않음.

**수정 방향:** 서버의 파싱 로직을 `{ option_text: string }[]` 형태로 수정하거나, 클라이언트에서 `string[]`으로 변환하여 전송.

## 🔴 P1 — MVP 공개 시 즉각적인 운영 문제를 일으키는 결함

### 2. 닉네임 중복 허용 (DB 레벨)

`users` 테이블에 `nickname`에 UNIQUE 제약이 없다. 동명이인이 생기면 멘션·검색 등 이후 기능 추가 시 혼란 발생. 지금 스키마를 바꾸는 게 훨씬 저비용.

```sql
-- 현재 스키마에 없음
ALTER TABLE public.users ADD CONSTRAINT users_nickname_unique UNIQUE (nickname);
```

> 단, 중복 닉네임이 이미 존재할 경우 회원가입 에러가 발생하므로, 프론트 측 실시간 중복 확인 UI도 함께 필요.

### 3. 비밀번호 재설정 기능 없음

로그인 페이지에 "비밀번호 찾기" 링크와 Supabase `resetPasswordForEmail` 연동이 없다. 이메일 인증 기반 서비스에서 비밀번호를 잊으면 계정 접근 방법이 없다. DAU가 생기기 시작하면 빠르게 CS 이슈가 된다.

### 4. 콘텐츠 신고 기능 없음

부적절한 게시글(혐오, 음란, 개인정보 포함 등)을 신고하는 수단이 없다. 유저 공개 시 즉시 운영 리스크. 최소한 신고 버튼과 어드민 확인 테이블이 있어야 한다.

### 5. 법적 필수 문서 없음

- 서비스 이용약관
- 개인정보처리방침

한국 기준 정보통신망법 상 개인정보를 수집(이메일, IP)하는 서비스는 개인정보처리방침 고지가 법적 의무다. 스토어 출시 시에도 필수.

### 6. 외부 이미지 URL 입력 UX 불량

현재 이미지 추가 방식이 "외부 URL 직접 입력"인데, 일반 유저는 이미지 URL을 모른다. `next.config.mjs`의 `remotePatterns`도 Supabase와 Google 이미지만 허용하므로 임의 외부 URL은 Next.js Image 컴포넌트에서 차단된다.

**선택지:**
- (A) 이미지 URL 입력 필드를 아예 제거하고 MVP는 텍스트 전용으로 시작
- (B) Supabase Storage 이미지 직접 업로드 구현

MVP에서는 (A)가 현실적. "곧 지원 예정" 문구와 함께 사용하기 어려운 기능을 두는 것은 신뢰성 저하.

## 🟡 P2 — UX 품질 문제 (공개 후 초기 이탈률에 직결)

### 7. 피드의 "종료됨" 탭에서 솔로몬의 선택이 표시되지 않음

홈 피드 서버 쿼리는 closed posts에 `vote_count`를 포함하지 않는다. `PostCard`의 `pickWinner()`는 `vote_count`가 없으면 "결과 확인하러 가기 →"만 표시한다.

결과: 종료된 카드가 모두 동일한 빈 상태로 보여 클릭 동기 약화. 결과를 피드에서 바로 볼 수 있어야 "솔로몬의 선택" 개념이 전달된다.

**수정 방향:** 홈 피드의 closed 쿼리에서 RPC `get_post_results`를 배치 호출하거나, 별도 컬럼(`solomon_choice_text`)을 posts 테이블에 캐싱.

### 8. 투표 후 피드로 돌아와도 "투표 완료" 상태 미반영

상세 페이지에서 투표 후 뒤로가기 하면 피드 카드는 여전히 "투표하기 →" 상태. SSR 피드 특성상 즉시 반영이 안 됨. 내가 투표했는지 피드에서 알 수 없어 혼란.

### 9. 소셜 로그인 없음 (진입 장벽)

모바일 환경에서 이메일 + 이메일 인증 링크 클릭 flow는 이탈률이 높다. Google/Kakao OAuth 중 하나라도 추가해야 가입 완료율이 유의미하게 높아진다. Supabase에서 OAuth 추가는 비교적 간단하나, MVP 범위 결정이 필요.

### 10. 첫 방문자에게 서비스 설명이 없음

홈 피드가 바로 게시글 목록으로 시작된다. "모두의 솔로몬이 무엇인지", "어떻게 쓰는지" 온보딩 흐름이 없다. 첫 방문 유저 기준 서비스 컨셉이 즉시 이해되기 어렵다.

### 11. 게시글 삭제 기능 없음

작성자가 본인 게시글을 삭제하는 UI가 없다. DB의 RLS에 delete 정책은 있으나(`posts_delete_owner`) 프론트 연동이 없다. 잘못 작성한 게시글을 수정·삭제할 수 없으면 유저가 불안해한다.

### 12. 투표 총 참여자 수 미표시

종료된 게시글에서 결과 바(ResultBar)에 퍼센트와 막대는 있지만, "총 N명 참여" 숫자가 없다. 투표 신뢰도와 화제성을 나타내는 핵심 정보.

### 13. 피드 무한 스크롤 없음

현재 active 50개, closed 50개 하드 리밋. 콘텐츠가 쌓이면 오래된 게시글 접근 불가. 인피니티 스크롤 또는 페이지네이션 필요.

### 14. 게시글 공유 기능 없음

상세 페이지에 공유 버튼이 없다. 모바일 Web Share API 연동과 OG 메타태그가 없으면 SNS 바이럴 전파가 차단된다. 커뮤니티형 서비스에서 공유는 핵심 성장 채널.

## 🟡 P3 — 운영/확장성 이슈

### 15. Vercel Cron 설정 확인 필요

`vercel.json`이 존재하지만 cron 설정이 없거나 불완전할 경우, 게시글이 자동으로 closed 처리되지 않아 24시간 룰이 작동하지 않는다. 배포 전 Vercel 대시보드에서 cron 작동 여부를 반드시 검증해야 한다.

```json
// vercel.json에 있어야 하는 최소 설정
{
  "crons": [
    {
      "path": "/api/cron/close-expired",
      "schedule": "*/5 * * * *"
    }
  ]
}
```

### 16. 솔로몬 지수 초기 UX 문제

새 가입자는 투표 기록이 0이라 지수 0%, 칭호 "새내기 시민"으로 시작. 지수 0% 표시는 심리적으로 부정적. 투표를 충분히 하지 않았을 때는 지수를 숫자로 표기하지 않고 "아직 투표 기록 없음" 상태로 처리하는 것이 더 낫다.

### 17. 게시글 상세 페이지가 클라이언트 렌더링 전용

`app/post/[id]/page.tsx`가 `"use client"`로 되어 있어 `useEffect`로 API 호출. SSR이 없어 SEO와 초기 로딩 성능에 불리하다. 특히 공유 링크를 통해 유입된 사용자에게 OG 메타태그가 없어 SNS 미리보기가 불가능.

### 18. 에러 화면이 최소화되어 있음

`error.tsx`와 `global-error.tsx`가 있지만 실제 구현 내용 확인 필요. 유저가 에러를 만났을 때 다음 행동을 안내하는 명확한 메시지가 있어야 한다.

### 19. Capacitor 네이티브 앱 의존성의 복잡도

`package.json`에 `@capacitor` 패키지들이 포함되어 있고 `android/`, `ios/` 디렉토리가 존재한다. MVP 목표가 웹인지 네이티브 앱인지 불명확하면, 두 플랫폼 모두 중도반단 상태가 된다. 배포 전략을 명확히 결정해야 한다.

## 잘 구현된 부분 (유지)

- **블라인드 룰 이중 검증:** 서버 RLS + API 레이어 + 클라이언트 레이어 3단계로 잘 구현됨.
- **DB 트리거 기반 솔로몬 지수:** 솔로몬 지수 갱신 로직이 DB 트리거로 원자적 처리됨. 정확도 높음.
- **Race condition 대응:** votes INSERT 시 UNIQUE constraint 위반(23505)과 RLS 거절(42501)을 분리 처리.
- **스켈레톤 로딩 UI:** 게시글 상세의 로딩 스켈레톤 구현 양호.
- **낙관적 투표 업데이트:** useVote 훅의 낙관적 업데이트 + 실패 롤백 구조.
- **Cron + RPC 분리:** 만료 처리 로직이 DB 함수로 캡슐화되어 재사용 가능.

## 결론

**공개 불가 사유:** 게시글 작성 자체가 동작하지 않는 P0 버그 존재.

**최소 공개 조건:** Phase 0 태스크 7개를 완료해야 실사용자에게 공개할 수 있는 최소 수준. 신고 기능 없이 공개하면 운영 리스크가 크므로 Phase 1의 신고 기능도 가능하면 Phase 0에 포함할 것을 권고.

**핵심 판단:** 기술 구조는 탄탄하다. 남은 것은 실사용자 관점의 완성도 문제다.
